# cf-task:align

从一句话需求出发，通过结构化对话产出设计简报（`.design.md`），为 `cf-task:plan` 提供足够丰富的输入。

## 输入

- `/cf-task:align "需求描述"` — 从需求描述新建设计简报
- `/cf-task:align` — 交互式，先问用户要做什么
- `/cf-task:align <.design.md 路径>` — 恢复已有草稿继续讨论

## 执行步骤

### 1. 确定模式

- 如果参数是已存在的 `.design.md` 文件路径 → **恢复模式**：Read 读取文件，展示当前内容，询问用户要调整哪些部分，进入 Step 3 继续讨论
- 如果参数是文本字符串 → **新建模式**：以该文本作为初始需求
- 如果无参数 → **交互模式**：询问用户"要实现什么功能？"

### 2. 代码库上下文扫描

快速了解项目现状，为后续建议提供依据：

1. 用 Glob 检查 `.code-flow/specs/*/_map.md` 是否存在，若有则 Read 读取（了解项目结构）
2. 检测技术栈：Read `package.json` / `requirements.txt` / `go.mod` / `pyproject.toml` 等（取存在的）
3. 用 Glob 扫描与需求相关的现有代码目录，了解已有模式

目的：**让后续建议基于实际代码库**，而非泛泛的通用建议。

### 2.5. 识别需求类型，选择维度

根据需求描述和代码库上下文，判断本次工作的性质，选择适用的讨论维度。

**核心维度（所有类型必选）**：
- 目标与边界
- 技术方案
- 约束条件
- 验收标准

**扩展维度（按需选择，不适用则跳过）**：
- 数据模型 — 涉及数据库/存储变更时
- 接口设计 — 涉及 API / CLI / 函数入口变更时

常见需求类型与维度匹配：

| 需求类型 | 数据模型 | 接口设计 | 技术方案侧重 |
|----------|---------|---------|-------------|
| 功能开发（含 API） | ✓ | ✓ API | 组件设计、第三方库选型 |
| 功能开发（纯后台） | 视情况 | 视情况 | 模块结构、算法/流程、数据流 |
| CLI 命令/脚本 | — | ✓ CLI | 命令结构、参数、输出格式 |
| 重构 | — | — | 变更范围、迁移路径、兼容性 |
| 性能优化 | — | — | 瓶颈分析、优化策略、度量指标 |
| 定时任务/队列 | 视情况 | — | 调度策略、失败处理、幂等性 |
| 日志/监控 | — | — | 采集点、指标定义、告警规则 |
| 构建/CI | — | — | 流水线设计、触发条件、产物 |
| 缺陷修复 | 视情况 | 视情况 | 根因分析、修复方案、回归范围 |

> 上表仅供参考，AI 应根据实际需求灵活判断，不机械套用。

### 3. 交互式细化

围绕选中的维度逐步推进，**每轮提 2-3 个问题**（不要一次性抛 10 个问题）：

**目标与边界**（必选）
- 解决什么问题？为谁？
- 做什么？（核心功能列表）
- 不做什么？（明确排除项）

**数据模型**（涉及数据库/存储变更时）
- 需要哪些表/集合？
- 字段、类型、约束、索引？
- 与现有模型的关系？（外键、关联）

**接口设计**（涉及 API / CLI / 函数入口时）

API 场景：
- 哪些 endpoint？请求/响应格式？错误码？

CLI 场景：
- 命令名称和子命令？参数和 flag？输出格式？

内部函数入口场景：
- 公开接口签名？输入输出类型？错误处理约定？

**技术方案**（必选，内容随需求类型变化）
- 功能开发：关键组件、模块结构、第三方库选型、与现有架构集成方式
- 重构：变更范围、分步迁移路径、向后兼容策略
- 性能优化：瓶颈定位、优化策略、性能度量方式
- 定时任务：调度策略、失败重试、幂等性保证
- 日志/监控：采集点、指标定义、告警阈值
- 构建/CI：流水线阶段、触发条件、产物和部署
- 缺陷修复：根因分析、修复方案、回归测试范围

**约束条件**（必选）
- 技术栈、性能要求、兼容性、时间线等

**验收标准**（必选）
- 怎么判断做完了？
- 可验证的条件列表

**交互原则**：
- 基于 Step 2 扫描到的代码库上下文**提出具体默认建议**，而非空泛提问
- 每轮回答后更新理解，追问细节或推进到下一个维度
- 涉及数据库和接口的维度**必须具体到表结构和请求/响应 schema**
- 不涉及数据库和接口时，**不要强制展开这些维度**
- 用户说"你定"或"随意"时，根据代码库上下文选择最合理的默认方案，明确告知用户采用了什么
- 输出语言默认与用户需求语言保持一致；若用户未明确指定语言，默认使用中文生成完整设计简报（标题、正文、表头均为中文）
- 仅当用户明确要求英文或项目已有强约定要求英文文档时，才输出英文；避免中英混杂

**中断恢复**：当任意两个维度已有实质内容时，立即写入草稿文件（Step 5 的写入逻辑，文件路径提前确定）。后续每轮对话更新草稿。这样即使会话中断，进度不会丢失。

### 4. 展示设计简报草稿

选中的维度都有实质内容后，展示设计简报。**只包含适用的章节，不生成空章节**。

#### 完整模板（功能开发含 API + DB 场景）

```markdown
# 设计简报：<模块名称>

## 目标
- <目标 1>
- <目标 2>

## 非目标
- <排除项 1>（原因）

## 数据模型设计

### <表名> 表
| 字段 | 类型 | 约束 | 说明 |
|--------|------|-------------|-------|
| ... | ... | ... | ... |

### 索引
- <索引定义>

## 接口设计

### <METHOD> <path>
- 请求: `{ field: type }`
- 响应 <status>: `{ field: type }`
- 错误 <status>: <错误描述>

## 技术决策
- <决策 1>: <选择>（原因）

## 约束条件
- <约束 1>

## 验收标准
- [ ] <可验证的验收条件 1>
```

#### 精简模板（纯后台/重构/优化等不涉及 API+DB 的场景）

```markdown
# 设计简报：<模块名称>

## 目标
- <目标 1>

## 非目标
- <排除项 1>（原因）

## 技术方案
<根据需求类型填写，示例：>

### 变更范围
- <涉及的模块/文件>

### 实现方案
- <步骤 1>
- <步骤 2>

### 关键决策
- <决策 1>: <选择>（原因）

## 约束条件
- <约束 1>

## 验收标准
- [ ] <可验证的验收条件 1>
```

#### 各场景 Technical Approach 示例

**CLI 命令**：
```markdown
## CLI 设计

### 命令结构
- `cf-task-align <requirement>` — 从需求新建
- `cf-task-align --resume <path>` — 恢复草稿

### 参数
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| requirement | string | 否 | 需求描述 |
| --resume | path | 否 | 恢复草稿路径 |

### 输出格式
- 成功: 设计简报路径
- 失败: 错误信息 + 退出码
```

**定时任务**：
```markdown
## 技术方案

### 调度策略
- 执行频率: 每 5 分钟
- 触发方式: cron

### 处理流程
1. 查询待处理队列
2. 批量处理（每批 100 条）
3. 更新状态

### 失败处理
- 重试策略: 指数退避，最多 3 次
- 幂等性: 基于 idempotency_key 去重
```

**重构**：
```markdown
## 技术方案

### 变更范围
- src/services/auth.js → 拆分为 auth-token.js + auth-password.js
- src/middleware/auth.js → 更新 import 路径

### 迁移路径
1. 创建新文件，复制逻辑
2. 更新所有 import
3. 删除旧文件
4. 验证测试通过

### 兼容性
- 对外接口不变（export 保持一致）
```

展示后询问用户：**确认写入？可以调整任何部分。**

### 5. 写入文件

用户确认后：

1. 从目标推导文件名（kebab-case），如 `user-auth`
2. 按当前日期创建目录：`.code-flow/tasks/<YYYY-MM-DD>/`
3. 用 Write 写入 `.code-flow/tasks/<YYYY-MM-DD>/<name>.design.md`

### 6. 输出摘要

```
已生成设计简报: .code-flow/tasks/<日期>/<name>.design.md

内容:
  - <按实际包含的章节列出>
  - 验收标准: N 项

下一步:
  - 审阅/编辑: 直接打开 .code-flow/tasks/<日期>/<name>.design.md
  - 标注问题: 在文件中添加 #NOTES，然后 /cf-task:note <name>.design
  - 生成任务: /cf-task:plan .code-flow/tasks/<日期>/<name>.design.md
```
